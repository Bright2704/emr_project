|--------|-----------|

| ประเภทเอกสาร | Requirements Specification |

---

## 1. บทนำ (Introduction)

### 1.1 วัตถุประสงค์ของเอกสาร
เอกสารนี้กำหนดความต้องการของระบบ EMR Scan Viewer ทั้งด้านธุรกิจ (Business), ด้านฟังก์ชัน (Functional) และด้านที่ไม่ใช่ฟังก์ชัน (Non-functional) เพื่อใช้เป็นฐานในการออกแบบและพัฒนาระบบ รวมถึงการสร้าง Prototype

### 1.2 ขอบเขตของระบบ (System Scope)
ระบบนี้เป็น Web Application สำหรับ **จัดเก็บและเรียกดูเอกสารสแกนของผู้ป่วย** โดยผูกกับข้อมูลผู้ป่วย (HN) และการเข้ารับบริการ (VN/AN) ระบบนี้ **ไม่ใช่ระบบ EMR เต็มรูปแบบ** และ **ไม่เชื่อมต่อกับระบบ HIS เดิมของโรงพยาบาล**

### 1.3 นิยามศัพท์ (Definitions)
| ศัพท์ | ความหมาย |
|------|----------|
| HN | Hospital Number — เลขประจำตัวผู้ป่วยของโรงพยาบาล (1 คน 1 เลข ตลอดชีวิต) |
| VN | Visit Number — เลขการเข้ารับบริการแบบผู้ป่วยนอก (OPD) |
| AN | Admission Number — เลขการเข้ารับบริการแบบผู้ป่วยใน (IPD / นอนโรงพยาบาล) |
| HIS | Hospital Information System — ระบบสารสนเทศหลักของโรงพยาบาล |
| eSCL | โปรโตคอลสแกนผ่านเครือข่ายแบบ driverless (Apple AirScan) |
| PDPA | พ.ร.บ.คุ้มครองข้อมูลส่วนบุคคล |
| Soft-delete | การลบแบบซ่อนข้อมูล แต่ยังเก็บไว้ในระบบเพื่อการตรวจสอบ |

## 2. ความต้องการเชิงธุรกิจ (Business Requirements)

| รหัส | ความต้องการ |
|------|-------------|
| BR-01 | ลดเวลาการค้นหาเอกสารผู้ป่วยจากการกองเอกสาร/ไฟล์กระจัดกระจาย ให้เรียกดูได้จากที่เดียว |
| BR-02 | บุคลากรเข้าถึงเอกสารผู้ป่วยได้รวดเร็ว โดยมีขั้นตอนเข้าสู่ระบบที่ไม่ซับซ้อน (low friction) |
| BR-03 | ลดความผิดพลาดในการจัดเก็บเอกสารผิดคน/ผิดครั้งของการรักษา |
| BR-04 | สามารถตรวจสอบย้อนหลังได้ว่าใครเข้าถึง/แก้ไขเอกสารใด (compliance) |
| BR-05 | ปกป้องข้อมูลส่วนบุคคลของผู้ป่วยตามมาตรฐาน PDPA |
| BR-06 | ระบบทำงานได้โดยไม่ต้องเชื่อมต่อกับ HIS เดิม เพื่อลดเวลาและความซับซ้อนในการ integrate |

## 3. ความต้องการเชิงฟังก์ชัน (Functional Requirements)

### 3.1 การยืนยันตัวตนและจัดการผู้ใช้ (Authentication & User Management)
| รหัส | ความต้องการ | ระดับ |
|------|-------------|-------|
| FR-A01 | แพทย์เข้าสู่ระบบด้วยเลขใบประกอบวิชาชีพเวชกรรม (เลข ว.) + PIN 6 หลัก | Must |
| FR-A02 | พยาบาลเข้าสู่ระบบด้วยเลขสภาการพยาบาล + PIN 6 หลัก | Must |
| FR-A03 | ระบบกำหนด Role ของผู้ใช้อัตโนมัติจากข้อมูลที่ Admin ตั้งไว้ (ผู้ใช้ไม่ต้องเลือก) | Must |
| FR-A04 | Admin โหลดรายชื่อบุคลากร (ชื่อ + เลขวิชาชีพ + role) ล่วงหน้าก่อนเปิดใช้งาน | Must |
| FR-A05 | ผู้ใช้ตั้ง PIN เองตอนเข้าใช้งานครั้งแรก พร้อมยืนยัน PIN และลงทะเบียนเบอร์/อีเมล | Must |
| FR-A06 | ใส่ PIN ผิด 1–4 ครั้งลองใหม่ได้ทันที, ครั้งที่ 5 เป็นต้นไปหน่วงเวลาเพิ่มขึ้น (progressive delay) | Must |
| FR-A07 | ระบบต้องไม่ล็อกบัญชีถาวร | Must |
| FR-A08 | ลืม PIN → ยืนยันตัวตนผ่าน OTP (เบอร์/อีเมล) → ตั้ง PIN ใหม่ | Must |
| FR-A09 | Admin / เวชระเบียน สร้าง/แก้ไข/ปิดบัญชี และ reset PIN ได้ | Must |
| FR-A10 | ระบบ auto-logout เมื่อไม่มีการใช้งานตามเวลาที่กำหนด (รองรับเครื่องใช้ร่วมกัน) | Must |

### 3.2 การจัดการผู้ป่วย (Patient Management)
| รหัส | ความต้องการ | ระดับ |
|------|-------------|-------|
| FR-P01 | สร้าง Patient Profile (ชื่อ, นามสกุล, วันเกิด, เลขบัตรประชาชน, HN) | Must |
| FR-P02 | ตรวจสอบเลขบัตรประชาชน/HN ก่อนสร้าง เพื่อกันสร้างซ้ำ และแจ้งเตือนหากมีอยู่แล้ว | Must |
| FR-P03 | ค้นหาผู้ป่วยด้วย HN หรือเลขบัตรประชาชน | Must |
| FR-P04 | แสดง list ผู้ป่วยที่ตรงกับคำค้น ให้เลือก (auto-select) | Must |
| FR-P05 | แก้ไขข้อมูลผู้ป่วย | Should |

### 3.3 การจัดการการเข้ารับบริการ (Visit Management)
| รหัส | ความต้องการ | ระดับ |
|------|-------------|-------|
| FR-V01 | สร้าง Visit (VN หรือ AN) ผูกกับ Patient | Must |
| FR-V02 | แสดงรายการ Visit ทั้งหมดของผู้ป่วยคนหนึ่ง | Must |
| FR-V03 | เลือก Visit เพื่อดูหรือเพิ่มเอกสาร | Must |

### 3.4 การสแกนและอัปโหลดเอกสาร (Document Scan & Upload)
| รหัส | ความต้องการ | ระดับ |
|------|-------------|-------|
| FR-S01 | แสดง preview ข้อมูลผู้ป่วย (ชื่อ-นามสกุล-วันเกิด) ให้ยืนยันก่อนเริ่มจัดเก็บ | Must |
| FR-S02 | สแกนเอกสารผ่านเครื่อง MFP บน LAN ด้วย eSCL (กดปุ่มในระบบ ไฟล์เข้า case อัตโนมัติ) | Must |
| FR-S03 | อัปโหลดไฟล์ด้วยตนเอง (PDF/PNG/JPG) เป็นทางเลือกสำรอง | Must |
| FR-S04 | เลือกหมวดหมู่เอกสารจาก dropdown ตอนจัดเก็บ | Must |
| FR-S05 | แสดง preview ปลายทาง (HN/VN/AN) ให้ยืนยันก่อนบันทึก | Must |

## 4. ความต้องการที่ไม่ใช่ฟังก์ชัน (Non-Functional Requirements)

| รหัส | ด้าน | ความต้องการ |
|------|------|-------------|
| NFR-01 | Security | ทุกการเชื่อมต่อผ่าน HTTPS/TLS |
| NFR-02 | Security | เข้ารหัสข้อมูลอ่อนไหว (เลขบัตรประชาชน) ทั้งตอนจัดเก็บและส่ง |
| NFR-03 | Security | ใช้ Role-Based Access Control (RBAC) |
| NFR-04 | Privacy | เลขบัตรประชาชนแสดงเต็มเฉพาะ role ที่มีสิทธิ์ อื่น ๆ เห็นแค่ 4 ตัวท้าย |
| NFR-05 | Compliance | สอดคล้องกับ PDPA สำหรับข้อมูลผู้ป่วย |
| NFR-06 | Usability | ขั้นตอน login ต้องเร็วและไม่ซับซ้อน เหมาะกับบุคลากรที่มีภาระงานสูง |
| NFR-07 | Performance | เปิดดูไฟล์เอกสารได้ภายในเวลาที่ยอมรับได้ (เป้าหมาย < 3 วินาทีต่อไฟล์) |
| NFR-08 | Compatibility | ทำงานบนเว็บเบราว์เซอร์หลัก (Chrome, Edge, Firefox, Safari) |
| NFR-09 | Maintainability | ออกแบบให้ดูแล/ขยายฟีเจอร์ (เช่น OCR เฟส 2) ได้ในภายหลัง |
| NFR-10 | Deployment | Software เท่านั้น — Server/Storage จัดหาโดยผู้ว่าจ้าง |

## 6. ข้อสมมติและข้อพึ่งพิง (Assumptions & Dependencies)
